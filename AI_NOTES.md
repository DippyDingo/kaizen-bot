## Текущее состояние проекта

### Telegram-бот

- Основной интерфейс работает как `single-message dashboard`.
- Основные разделы:
  - `Главная`
  - `Задачи`
  - `Календарь`
  - `Дневник`
  - `Здоровье`
  - `Статистика`
  - `Настройки / Профиль`
- Глобальная навигация идет через быстрые кнопки у поля ввода.
- Нижняя системная кнопка Telegram используется под `Web App` с текстом `Open`.
- Ссылка на активный dashboard хранится и в FSM state, и в БД.

### Backend

- Используется async `SQLAlchemy` + `Alembic`.
- Есть read-only `aiohttp` API foundation в `backend/api/`.
- Реализованы read-only endpoint'ы:
  - `GET /api/v1/dashboard`
  - `GET /api/v1/health`
  - `GET /api/v1/stats`

### Web App

- Есть multi-screen read-only Web App shell.
- Экранов сейчас три:
  - `dashboard`
  - `health`
  - `stats`
- Web App использует `X-Telegram-Init-Data` и не имеет dev-обхода по `telegram_id`.

### Здоровье

- Реализованы блоки:
  - вода
  - сон
  - тренировки
  - лекарства
  - состояние (`энергия / стресс`)
- Есть дневные и недельные summary-экраны.
- Есть периодическая статистика по здоровью.

### 2026-03-08 — Полный рефакторинг календаря лекарств

Что добавлено / изменено:
- Логика medication-календаря переписана вокруг единой модели статусов приема:
  - `taken` — прием подтвержден;
  - `pending` — прием еще не закрыт на текущий или будущий день;
  - `skipped` — прием пропущен на прошедший день или явно помечен старым логом `skipped`.
- Введен единый helper-слой для расчета статусов по дням и курсам:
  - нормализация логов по последнему состоянию `(course_id, scheduled_date)`;
  - единый расчет day-buckets для календаря, дневного списка и агрегатов.
- Календарная метка дня теперь считается по явным правилам:
  - `✅` только если все приемы дня закрыты как `taken`;
  - `✖️` если на прошедший день есть хотя бы один `skipped`;
  - `💊` если приемы есть, но день еще не завершен полностью (`pending`).
- Удаленные курсы не оставляют меток в medication-календаре.
- Дневной список лекарств теперь различает `pending` и `skipped`:
  - `pending` показывается как ожидание,
  - `skipped` как реальный пропуск.
- В payload health/API добавлены `pending`-счетчики по лекарствам.
- При снятии отметки `Выпил` notice теперь зависит от реального статуса после снятия:
  - `pending` для текущего/будущего дня,
  - `skipped` для прошедшего дня.

### 2026-03-08 — Исправление меток календаря лекарств

Что добавлено / изменено:
- Исправлена логика календарных меток во вкладке лекарств.
- Раньше день помечался как `✅ все выпито`, если количество отметок `taken` совпадало с рассчитанным числом приемов в агрегате для дня, что ломало сценарий с несколькими курсами и частично отмеченными приемами.
- Дополнительно исправлен более глубокий дефект: агрегация считала сырые `MedicationLog`, а не последнее состояние по ключу `(course_id, scheduled_date)`. Из-за этого старые записи могли давать ложный `✅`.
- Исправлен второй исторический дефект: календарь раньше считал только `is_active=True` курсы. Если курс позже отключали, прошлые дни пересчитывались как будто этого лекарства не существовало, и день мог ложно стать `✅`.
- После ручной проверки UX-ожидание уточнено: удаленный курс не должен оставлять календарную метку во вкладке лекарств. Поэтому исторический учет отключенных курсов для medication-календаря откатан обратно к пользовательской логике.
- Теперь логика разделена явно:
  - `✅` только если все запланированные приемы на день отмечены как выпитые;
  - `💊` если приемы на день есть, но день заполнен частично или это текущий/будущий день без полной отметки;
  - `✖️` только если это прошедший день с запланированными приемами без подтверждения.
- В helper'ах лекарств добавлена нормализация: для каждого дня и курса берется только последний лог.
- Для medication-календаря снова учитываются только активные курсы, потому что удаление курса должно убирать его из интерфейса пользователя.
- Добавлены regression-тесты:
  - на частично выполненный день и будущий день;
  - на конфликтующие дубли логов, где учитывается только последнее состояние.
  - на отключенный курс, который не должен оставлять метку в календаре.

Измененные файлы концептуально:
- `backend/services/health_parts/medication.py`
- `tests/test_health_service_medications.py`

## Текущие ограничения

- Live-проверка Telegram зависит от доступа к `api.telegram.org`.
- Быстрые кнопки чата (`ReplyKeyboard`) по ограничениям Telegram не появляются сами по себе до первого пользовательского update.
- После удаления истории Telegram удаляет carrier-сообщение клавиатуры; корректное восстановление идет через новый `/start`.
- Web App через временный tunnel подходит только для разработки и ручной проверки.
- В репозитории уже есть незакоммиченные локальные изменения вне этого файла, включая `README.md`.
